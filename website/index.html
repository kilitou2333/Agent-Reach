为什么需要 Agent Reach？

AI Agent 能写代码、改文档、管项目——但你让它去网上找点东西，它就抓瞎了。

📺"看看这个 YouTube 教程讲了什么" → 拿不到字幕
🐦"搜一下推特上怎么评价这产品" → Twitter API 要付费
📖"去 Reddit 看有没有人遇到同样 bug" → 403 被封
📕"看看小红书上这个品口碑" → 必须登录才打得开
🔍"网上搜下最新 LLM 框架对比" → 没有好用的免费搜索
🌐"看看这个网页写了啥" → 抓回一堆 HTML 标签

设计理念：一个能力层，不是又一个工具

Agent Reach 比任何具体实现高一层——负责选型、安装、体检、路由，不负责底层读取本身。读取由 Agent 直接调用上游工具完成，没有包装层。换接入方式 = 调整列表顺序，不是重写代码。

channels/ ├── web.py → Jina Reader ├── twitter.py → twitter-cli ▸ OpenCLI ▸ bird ├── youtube.py → yt-dlp ├── github.py → gh CLI ├── bilibili.py → bili-cli ▸ OpenCLI ▸ 搜索 API ├── reddit.py → OpenCLI ▸ rdt-cli ├── xiaohongshu.py→ OpenCLI ▸ xiaohongshu-mcp ▸ xhs-cli ├── linkedin.py → linkedin-mcp ▸ Jina Reader └── rss.py → feedparser

安全性

在设计上重视安全：凭据本地存储、安全模式、完全开源、Dry Run 预览。

🔒

凭据本地存储

Cookie / Token 只存本机 ~/.agent-reach/config.yaml，权限 600，不上传。

🛡️

安全模式

--safe 不会自动改系统，只列出需要什么，由你决定。

👀

完全开源

代码透明可审查，所有依赖也是开源项目。

🔍

Dry Run

--dry-run 预览所有操作，不做任何改动。

⚠️ 用 Cookie 登录的平台（Twitter、小红书等）存在被平台检测封号的风险，请使用专用小号，不要用主账号。
